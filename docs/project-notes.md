# Правила разработки

## 1. Ядро проекта

Мы строим приложение изнутри наружу.

Сначала создаем доменную модель.

Потом бизнес-логику.

И только после этого подключаем внешние API.

Порядок разработки:

Domain Models
↓

Services

↓

CLI

↓

Spotify API

↓

YouTube Music API

---

## 2. AI — второй разработчик

Во время разработки мы обсуждаем архитектурные решения.

Перед написанием нового компонента отвечаем на вопросы:

- Зачем он нужен?
- Кто будет его использовать?
- Не нарушает ли он SRP?
- Можно ли сделать проще?

---

## 3. Один коммит — одна законченная задача

Каждый коммит должен:

- решать одну задачу;
- собираться;
- быть законченным;
- иметь понятное название.

Используем Conventional Commits.

Например:

feat(domain): add Track model

---

## 4. Документация развивается вместе с проектом


README

- Что это за проект.

Architecture
 
 - Как устроен проект.

ADR

- Почему было принято конкретное архитектурное решение.

---

## 5. YAGNI - You are not gonna need It

Не добавляем функциональность заранее.

Если сущность пока нигде не используется — она не появляется в проекте.

---

## 6. Сначала ответственность, потом интерфейс

Перед созданием любого интерфейса сначала определяем ответственность компонента.

Отвечаем на вопросы:

- За что отвечает компонент?
- Кто будет его использовать?
- Какие обязанности он выполняет?

Только после этого проектируем интерфейс.

Интерфейс должен описывать одну роль, а не объединять множество несвязанных обязанностей.

---

## 7. Не проектировать под текущую реализацию

Доменные модели описывают предметную область, а не особенности конкретной платформы или алгоритма.

Например:

Track хранит информацию о песне.

Track не должен содержать:

- OAuth Token
- HTTP Client
- SpotifyID (пока это не требуется)
- SearchQuery
- MatchedVideoID

Если новое поле появляется только потому, что "может пригодиться", то его не добавляем.

Сначала возникает необходимость — потом меняется модель.

---

## 8. Не проектировать то, чего еще не существует

Не добавляем сущности, поля или компоненты заранее.

Например:

- SpotifyID
- Cache
- AppleMusicProvider

появятся только тогда, когда действительно возникнет необходимость.

Сначала появляется задача — потом архитектурное решение.

---

## 9. Понимать причину каждого решения

Перед добавлением новой сущности, интерфейса или сервиса отвечаем на вопрос:

Почему именно так?

Важно понимать не только "как", но и "почему".

Во время разработки обсуждаем архитектурные решения так, как это происходит на настоящем code review.

# Предметная область

## Track

Track представляет песню внутри нашего приложения.

Это единая модель, с которой работает бизнес-логика.

Provider самостоятельно преобразует данные платформы в Track.

Track не знает ничего о Spotify, YouTube Music или других сервисах.

Для MVP Track должен содержать только информацию о песне.

Предварительно:

- Title
- Artists
- Album
- Duration
- ISRC (если доступен)

> Пока не добавляем идентификаторы платформ. Если они понадобятся в будущем, вернемся к этому решению.

---

## Playlist

Playlist представляет плейлист внутри приложения.

Предварительно содержит:

- Name
- Description
- Tracks
- Visibility

---

## User

Пока модель User отсутствует.

После получения OAuth Token приложение практически не взаимодействует с пользователем.

Если позже User действительно понадобится — добавим.

# Основные сервисы

## Provider

Provider представляет конкретную музыкальную платформу.

Например:

- SpotifyProvider
- YTMusicProvider

Provider самостоятельно:

- взаимодействует с API платформы;
- преобразует ответы API в доменные модели;
- скрывает особенности реализации платформы.

TransferService работает только с доменными моделями и не знает, как именно Provider получает данные.

---

## TransferService

TransferService отвечает только за процесс переноса.

Он ничего не знает про:

- HTTP;
- OAuth;
- JSON;
- особенности Spotify API;
- особенности YouTube Music API.

Он работает исключительно через Provider.

---

## Matcher

Matcher отвечает за поиск наиболее подходящей песни.

Например:

Spotify:

Linkin Park — Numb

YouTube Music:

Linkin Park — Numb (Official Video)

или

Numb (Remastered)

Matcher определяет, является ли найденная песня подходящей.

Алгоритм Matcher можно улучшать независимо от остальной системы.

> Для MVP Matcher не реализуется.
> На первом этапе используется простой поиск.
